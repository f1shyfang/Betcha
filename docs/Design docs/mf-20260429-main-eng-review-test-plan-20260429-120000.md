## Affected Pages/Routes
- `/` — landing / quick-create market flow (sanity)
- `/groups/[id]` — group dashboard: markets list, invite link
- `/markets/[id]` — market detail, place prediction, view leaderboard
- `/api/markets` — create/list markets
- `/api/markets/:id/predictions` — place/update prediction
- `/api/markets/:id/resolve` — resolve market
- `/api/groups/:id/invites` — create invite token

## Key Interactions to Verify
- Create group → issue invite → friend uses invite → join group
- Create market in group → friends place predictions → creator resolves → ledger updates
- Creator resolves unfairly → dispute opened → vote resolves outcome
- Leaderboard/score updates show consistent totals across views

## Edge Cases
- Double-submit prediction (network retry / rapid clicks)
- Concurrent resolves (two users attempt to resolve simultaneously)
- Stale page: user places prediction after market already resolved
- Network timeouts during resolve / ledger write
- Invite token reuse or expired token usage

## Critical Paths
- Market creation → prediction → resolution → ledger delta (end-to-end)
- Dispute workflow: resolution → dispute open → votes → final resolution

## Tests to Add
- `tests/e2e/market_create_predict_resolve.test.ts` (E2E): complete end-to-end happy path. CRITICAL.
- `tests/api/predictions.test.ts` (unit): idempotent place/update behavior, duplicate submission.
- `tests/api/resolve.test.ts` (unit+integration): idempotency + concurrent resolve attempts.
- `tests/workflows/dispute_vote.test.ts` (E2E): dispute open → votes → majority outcome; tie path flagged as manual/pilot ops.
- `tests/api/invites.test.ts` (unit): expired token, reused token negative cases.

## Failure modes (one per critical path)
- Concurrent resolves: two requests succeed → inconsistent ledger; mitigation: DB transaction + `market.state` check + unique constraint on `resolutions.market_id` (test covers).
- Prediction after resolve: leads to orphan prediction -> mitigate: reject predictions when market.state != 'open' (test covers).
- Network timeout during ledger writes: partial writes -> mitigation: single DB transaction for resolution and ledger updates (test + integration).
